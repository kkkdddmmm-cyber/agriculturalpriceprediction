# 농산물 가격 예측 시스템 — 클라우드 배포 가이드 (방식 A)

VM 없이 **GitHub Actions + Streamlit Community Cloud** 로 무료 운영하는 방법입니다.

- 매일 데이터 수집 + 예측 → **GitHub Actions** 가 실행 (VM 불필요)
- 대시보드 웹 화면 → **Streamlit Community Cloud** 가 호스팅
- 데이터(CSV/JSON/모델/이미지) 저장소 → **GitHub 저장소** 자체

---

## 0. 최종 저장소 구조

```
agricultural-price-prediction/
├── app.py
├── api.py
├── 최종_모델.py
├── requirements.txt
├── .gitignore
├── .env.example
├── 보관/                     ← 배추·양파·고구마 CSV
├── models/                   ← joblib, performance.json, prediction_log.json, SHAP png
└── .github/
    └── workflows/
        └── daily-update.yml  ← 매일 23시 자동 실행 정의
```

> ⚠️ 폴더 이름이 한글(`보관`)이어도 동작하지만, 혹시 문제가 생기면 `data` 등 영문으로
> 바꾸고 코드의 경로 문자열도 함께 바꾸면 더 안전합니다.

## 1. 코드에서 이미 고쳐둔 것 (그대로 쓰면 됨)

- **경로**: `app.py`, `api.py`, `최종_모델.py` 모두 절대경로 →
  `os.path.dirname(os.path.abspath(__file__))` 기준 **상대경로** 로 변경.
  (로컬·클라우드 어디서든 "스크립트가 있는 폴더" 기준으로 동작)
- **폰트**: `최종_모델.py` 의 윈도우 전용 `Malgun Gothic` →
  리눅스에서는 NanumGothic 을 자동 다운로드하도록 변경 (SHAP 한글 안 깨짐).
- **비밀 키**: `api.py` 의 키/비밀번호를 코드에서 제거하고 **환경변수/.env** 에서 읽도록 변경.
- **실제값 백필**: `최종_모델.py` 에 매 실행 시 CSV 실제값으로 `null` 을 채우는 로직 추가.

## 2. 깃허브에 올리기

1. 깃허브에서 새 저장소 생성 (private 권장 — 데이터/모델 보호).
2. 로컬에서:

   ```bash
   git init
   git add .
   git commit -m "first commit"
   git branch -M main
   git remote add origin https://github.com/사용자명/저장소명.git
   git push -u origin main
   ```

   > `.gitignore` 가 `.env` 를 막아주므로 **비밀 키는 올라가지 않습니다.** 꼭 확인하세요.

## 3. GitHub Secrets 등록 (자동 실행이 키를 쓰도록)

저장소 → **Settings → Secrets and variables → Actions → New repository secret**
아래 6개를 **각각** 등록 (이름은 정확히 동일하게):

| 이름 | 값 |
|------|----|
| `KAMIS_KEY`    | KAMIS 키 |
| `KAMIS_ID`     | KAMIS ID |
| `WEATHER_KEY`  | 기상청 키 |
| `OPINET_KEY`   | 오피넷 키 |
| `GARAK_ID`     | 가락시장 ID |
| `GARAK_PASSWD` | 가락시장 비밀번호 |

> ❗ 예전에 키가 코드에 박힌 채 깃허브(특히 public)에 올라간 적이 있다면,
> 각 기관 사이트에서 **키를 재발급** 받는 것이 안전합니다. (노출된 키는 폐기)

---

## 4. 자동 실행 동작 방식

`.github/workflows/daily-update.yml` 이 매일 동작합니다.

- 실행 시각: **한국시간 23:00** (= cron `0 14 * * *`, UTC 14:00)
- 순서: ① `api.py` (실제값 수집) → ② `최종_모델.py` (백필 + 내일 예측) → ③ 바뀐 파일 커밋·푸시
- 수동 실행: Actions 탭 → daily-update → **Run workflow** 버튼

> cron 은 정확히 그 시각이 아니라 수 분~수십 분 늦을 수 있습니다(정상).
> 처음에는 **Run workflow** 로 한 번 수동 실행해서 로그가 초록불인지 확인하세요.

## 5. Streamlit Community Cloud 에 앱 띄우기

1. https://share.streamlit.io 접속 → 깃허브 계정으로 로그인.
2. **New app** → 저장소 / 브랜치(main) / 메인 파일(`app.py`) 선택.
3. 만약 `app.py` 가 직접 API 키를 쓴다면(현재는 안 씀) **Advanced settings → Secrets**
   에 키를 TOML 형식으로 입력. (지금 구조에선 불필요)
4. **Deploy** → 잠시 후 `https://....streamlit.app` 주소가 생성됩니다.

GitHub Actions 가 매일 데이터를 커밋하면, Streamlit 은 저장소 변경을 감지해
최신 차트를 자동 반영합니다. (즉시 반영이 안 보이면 앱 우측 상단 **Reboot**)

## 6. 로컬에서 테스트하기

```bash
pip install -r requirements.txt
cp .env.example .env       # .env 에 실제 키 입력
python api.py              # 데이터 수집
python "최종_모델.py"       # 백필 + 예측
streamlit run app.py       # 대시보드 확인
```

## 자주 막히는 곳

- **한글이 네모(□)로 보임** → 폰트 자동 다운로드가 막힌 경우. 보통 NanumGothic 다운로드로 해결됨.
- **Actions 가 push 에서 실패** → 워크플로의 `permissions: contents: write` 가 있는지 확인.
  (Settings → Actions → General → Workflow permissions = **Read and write** 로도 설정)
- **실제값이 계속 null** → 반드시 `api.py` 가 **먼저**, `최종_모델.py` 가 **나중**에 실행돼야 함.
  워크플로는 이미 그 순서로 되어 있음.
- **시간이 안 맞음** → cron 은 UTC. 한국 23시는 `0 14 * * *`.
- **저장소 용량 증가** → CSV 는 하루 한 줄이라 괜찮음. 모델/이미지가 크면 주기적으로 정리하거나
  Git LFS 사용 고려.
